## Code Quality

### Overall Assessment
The codebase is well-structured for an early-stage Next.js 16 App Router project. Separation of concerns is clear: `src/lib/` holds utilities, `src/app/api/` routes handle business logic, and layouts enforce auth at the page level. TypeScript is used consistently.

### Observations

**Positive:**
- ECPay `generateCheckMacValue` and `verifyECPayCallback` are cleanly separated and the algorithm is correctly implemented per ECPay spec (space→`+`, SHA256, uppercase).
- `escapeHtml` is applied to all form field names and values in `buildECPayForm`, preventing HTML injection into the auto-submit form.
- `bookings_insert_customer` RLS policy forces `customer_id = auth.uid()` at the DB level, preventing customer_id spoofing.
- The booking state machine `VALID_PROVIDER_TRANSITIONS` is explicit and enforced in the provider API.
- Financial calculations (`calculateFinancials`) are pure functions with no side effects and are easy to unit test.
- Admin layout (`src/app/(admin)/admin/layout.tsx`) and admin API routes both independently check `role === 'admin'`, providing defence-in-depth.

**Issues:**

1. **Widespread use of `supabase as any` (12+ occurrences)** — Every API route casts the typed Supabase client to `any` to suppress type errors, eliminating all type-safety guarantees on DB queries. This is a code smell that indicates the generated `Database` type in `src/types/database.ts` is incomplete or mismatched, rather than a deliberate design choice. File count: `src/app/api/bookings/route.ts:39`, `provider/services/route.ts:23,50`, `provider/bookings/[bookingId]/route.ts:54`, `admin/cleaners/[cleanerId]/route.ts:29`, `admin/payouts/[payoutId]/route.ts:30`, and others.

2. **`generateBookingNumber` uses `Math.random()` (LOW entropy)** — `src/lib/booking.ts:26-29`. The random suffix is only 4 base-36 characters (~1.7M possibilities). Under concurrent load, booking number collisions are possible. This is also not cryptographically random. A duplicate `booking_number` would cause a DB `UNIQUE` constraint violation and return a 500 error to the user with no retry.

3. **No input validation on numeric fields from the booking API** — `src/app/api/bookings/route.ts:32`. `areaSqm`, `scheduledDate`, `scheduledStartTime` are passed from user-controlled JSON body. `scheduledStartTime` is directly split and used in arithmetic at line 88 without checking format. A value like `"99:99"` would produce an invalid `scheduled_end_time` of `"105:39"` stored in the DB without error.

4. **`platform_settings` table has no RLS enabled** — `supabase/migrations/002_rls.sql` enables RLS on all listed tables, but `platform_settings` (defined in `001_schema.sql:218`) is not in the list. Any authenticated or anonymous user can read platform settings with anon key.

5. **Silent swallowed error in `server.ts`** — `src/lib/supabase/server.ts:18`: `} catch {}` silently swallows cookie-setting errors. This is copied from Supabase SSR boilerplate but should at minimum log the error in development.

6. **`requireCustomer` in `src/lib/auth.ts:19` does not verify role** — It only checks that a `profile` row exists. A user who registered as a `cleaner` but navigates to a customer-only page passes this guard.

## Security

### ECPay Integration

**`src/lib/ecpay.ts` + `src/app/api/ecpay/notify/route.ts`**

The notify webhook **correctly calls `verifyECPayCallback`** before processing any payment state change (line 24 of `notify/route.ts`). This is the most critical security check and it is in place.

`verifyECPayCallback` in `src/lib/ecpay.ts:134-144` correctly:
- Strips `CheckMacValue` from the params before computing the expected value
- Returns `false` when `CheckMacValue` is absent
- Does an uppercase comparison matching ECPay spec

No bypass of signature verification was found.

**One weakness:** The `hashKey` and `hashIv` are read with the non-null assertion operator (`!`) without runtime checks. If these env vars are undefined at runtime (misconfigured deploy), `generateCheckMacValue` will receive `undefined` cast to string — `"HashKey=undefined&..."` — and will compute a deterministic hash. An attacker who discovers this misconfiguration could craft a valid-looking MAC. Mitigation: add explicit env-var presence checks at startup or in the function.

### Supabase RLS

**`supabase/migrations/002_rls.sql`**

| Table | User isolation | Finding |
|---|---|---|
| profiles | `id = auth.uid()` | Correct |
| cleaners | `profile_id = auth.uid()` for writes | Correct |
| bookings | `customer_id = auth.uid() OR cleaner_id = auth_cleaner_id()` | Correct |
| payments | Subquery on bookings | Correct — cleaner cannot see payment amounts |
| reviews | `customer_id = auth.uid()` | Correct |
| payouts | `cleaner_id = auth_cleaner_id()` | Correct |
| notifications | `user_id = auth.uid()` | Correct |
| **platform_settings** | **No RLS** | **FINDING — see below** |

The `bookings_update` policy at line 58-60 allows both `customer_id = auth.uid()` AND `cleaner_id = auth_cleaner_id()` to update a booking. This means a customer could directly update `status`, `total_amount`, `cleaner_payout`, etc. via a direct Supabase client call. The API layer enforces correct transitions, but RLS does not restrict *which columns* can be updated. This is a meaningful attack surface if the anon key is exposed to the browser (it is, via `NEXT_PUBLIC_SUPABASE_ANON_KEY`).

The `reviews_update_own` policy at line 73 allows `cleaner_id = auth_cleaner_id()` to update a review. A cleaner could edit a customer's comment text (`comment` column) in addition to their own reply. This likely should be restricted to `cleaner_reply` and `cleaner_replied_at` only.

### Admin Routes

Both the UI layer (`src/app/(admin)/admin/layout.tsx:17`) and the API layer (`src/app/api/admin/cleaners/[cleanerId]/route.ts:13`, `src/app/api/admin/payouts/[payoutId]/route.ts:25`) independently verify `role === 'admin'`. The middleware at `src/middleware.ts` does **not** enforce route-level access control — it only calls `updateSession` to refresh the Supabase auth cookie. This means protection relies entirely on each layout and route handler checking the role. This is correct as long as every admin route and API route consistently checks — which they currently do — but there is no central enforcement to prevent a future developer from creating an admin API route without the check.

### SQL Injection / XSS

No raw SQL is used. All queries go through the Supabase PostgREST client with parameterised queries. No SQL injection surface found.

No user-controlled content is rendered via `dangerouslySetInnerHTML`. The `escapeHtml` function in `src/lib/ecpay.ts:146-152` is applied before interpolating into the auto-submit HTML form. No XSS surface found.

### Secrets

No hardcoded credentials found. All sensitive values reference `process.env.*`. The `raw_notify` ECPay payload is stored as JSONB in the `payments` table — this is acceptable for audit purposes and does not include secrets.

## Architecture

### Strengths
- The dual-path auth pattern (layout guard for UI, route handler guard for API) is correctly implemented in both admin and provider sections.
- The ECPay notify webhook correctly uses the **service role key** (bypassing RLS, which is appropriate for a server-to-server webhook with no user session) while all user-facing API routes correctly use the **anon key with cookie session**.
- Booking financials are snapshotted at booking creation time (columns: `total_amount`, `platform_commission`, `cleaner_payout`, `commission_rate`), which is correct — changes to service prices after booking do not affect existing bookings.
- The `booking_status_logs` table provides a full audit trail of every state transition.
- The `payments.raw_notify` JSONB column stores the full ECPay callback for debugging.

### Weaknesses
- **No idempotency guard on the ECPay notify webhook.** `notify/route.ts:44-47` checks `booking.status !== 'pending_payment'` before processing — this is the correct guard and prevents double-processing. However, it does not use a database-level lock or transaction. Under a race condition where ECPay sends two concurrent notify calls for the same trade, both could read `status = 'pending_payment'` before either has written `status = 'paid'`, resulting in duplicate payment inserts and duplicate notifications. A `SELECT FOR UPDATE` or upsert-with-conflict would be safer.
- **No transaction wrapping in notify webhook.** The three write operations in `notify/route.ts` (update booking, update payment, insert log) are three independent Supabase calls. If the second or third call fails, the database is left in a partially-updated state (e.g., booking marked `paid` but no payment record updated).
- **Provider layout does not redirect unapproved cleaners.** `src/app/(provider)/provider/layout.tsx:78-90` shows an onboarding CTA if `cleaner` is null (no record at all), but if a cleaner's status is `suspended`, they see the full dashboard UI — only the sidebar badge changes. Provider API routes do not verify `cleaner.status === 'approved'`, meaning a suspended cleaner can still confirm bookings via API.
- **Conflict detection in booking API is date-only, not time-aware.** `src/app/api/bookings/route.ts:71-82` checks for any existing booking on the same `scheduled_date`. This means a cleaner can only accept one booking per calendar day, even if two bookings are for different times. This is probably overly restrictive and should be a time-overlap check.
- **Middleware is a pure session refresh with no route protection.** `src/middleware.ts` delegates entirely to `updateSession`. There is no centralized protection of `/admin/*` or `/provider/*` routes at the middleware level. This creates a risk of future route additions lacking guards.

## Issues Found

### CRITICAL

None found that are exploitable in isolation. The most impactful combination is described below as MAJOR.

---

### MAJOR

**[MAJOR-1] Overly permissive `bookings_update` RLS policy allows customers to self-modify financial fields**

File: `supabase/migrations/002_rls.sql:58-60`

```sql
CREATE POLICY "bookings_update" ON bookings FOR UPDATE USING (
  customer_id = auth.uid() OR cleaner_id = auth_cleaner_id() OR auth_user_role() = 'admin'
);
```

The policy grants full row update rights without column restrictions. A customer can issue a direct Supabase client call (using the public anon key) to update `total_amount`, `cleaner_payout`, `platform_commission`, or `status` on their own booking. The API layer enforces this correctly, but RLS does not. Supabase supports column-level security via `WITH CHECK` expressions or Postgres column privileges.

Fix: Add a `WITH CHECK` clause that restricts which fields customers can update, or use Postgres column-level `GRANT UPDATE (address, notes, ...) ON bookings TO authenticated` instead of full row access.

---

**[MAJOR-2] `platform_settings` table has no RLS and no INSERT/UPDATE policy**

File: `supabase/migrations/002_rls.sql` (by omission) and `supabase/migrations/001_schema.sql:217-224`

The `platform_settings` table is created in `001_schema.sql` but is not listed in the `ALTER TABLE ... ENABLE ROW LEVEL SECURITY` block in `002_rls.sql`. In Supabase, tables without RLS enabled are accessible to all roles including `anon`. Any user or script with the public anon key can read — and potentially write — platform configuration keys.

Fix: Add `ALTER TABLE platform_settings ENABLE ROW LEVEL SECURITY;` and a policy: `CREATE POLICY "settings_admin_only" ON platform_settings FOR ALL USING (auth_user_role() = 'admin');`

---

**[MAJOR-3] ECPay notify webhook has no transaction — partial update risk**

File: `src/app/api/ecpay/notify/route.ts:54-76`

Three separate DB writes (update booking, update payment, insert log) are made without a transaction. If the payment record update (line 62-68) fails after the booking status is already set to `'paid'` (line 54-57), the booking shows as paid but the payment record is never updated. Similarly, the `booking_status_logs` insert is a best-effort call with no error handling at lines 71-76.

Fix: Use Supabase's `rpc()` to call a PostgreSQL function that wraps all three writes in a `BEGIN/COMMIT` transaction block, or at minimum check each write's error and return `0|InternalError` if any fails, allowing ECPay to retry.

---

**[MAJOR-4] Suspended cleaners are not blocked from provider API actions**

File: `src/app/api/provider/bookings/[bookingId]/route.ts:29-37`  
File: `src/app/api/provider/services/route.ts:39-40`

Both provider API routes verify that a cleaner row exists (via `profile_id = user.id`) but do not check `cleaner.status`. A cleaner with `status = 'suspended'` can:
- Accept and advance bookings through the state machine
- Create new services

Fix: After fetching the cleaner row, add: `if (cleaner.status === 'suspended') return NextResponse.json({ error: 'Account suspended' }, { status: 403 })`

---

### MINOR

**[MINOR-1] `reviews_update_own` policy allows cleaners to edit customer comment text**

File: `supabase/migrations/002_rls.sql:73`

```sql
CREATE POLICY "reviews_update_own" ON reviews FOR UPDATE USING (
  customer_id = auth.uid() OR cleaner_id = auth_cleaner_id() OR auth_user_role() = 'admin'
);
```

A cleaner can update the `comment` column (the customer's review text) via direct Supabase client call. Intended behaviour is for cleaners to only update `cleaner_reply` and `cleaner_replied_at`.

Fix: Split into two separate policies — one for `customer_id = auth.uid()` covering all columns, one for `cleaner_id = auth_cleaner_id()` restricted to `cleaner_reply, cleaner_replied_at` via column-level privilege or a `WITH CHECK` that leaves `comment` unchanged.

---

**[MINOR-2] `generateBookingNumber` uses low-entropy `Math.random()`**

File: `src/lib/booking.ts:25-30`

`Math.random()` is not cryptographically random. The 4-character base-36 suffix gives ~1.7 million combinations. Under concurrent load (e.g., a flash sale), collisions will trigger a 500 error with no user-visible retry. ECPay also requires `MerchantTradeNo` to be unique per merchant account — a collision would result in a failed payment form submission.

Fix: Replace the random portion with `crypto.randomBytes(4).toString('hex')` for 4 billion combinations, or use a UUIDv4 truncated to the allowed length.

---

**[MINOR-3] `scheduledStartTime` input not validated before arithmetic**

File: `src/app/api/bookings/route.ts:88-92`

```typescript
const [hours, minutes] = scheduledStartTime.split(':').map(Number)
const endMinutes = hours * 60 + minutes + Math.round(service.duration_hours * 60)
```

If `scheduledStartTime` is not in `HH:MM` format (e.g., `"abc"`, `"9"`, `"99:99"`), `hours` and `minutes` will be `NaN`, producing a `scheduled_end_time` of `"NaN:NaN"`. This passes the DB insert without error since the column is `TIME` type and Supabase/Postgres would reject it — but the error message returned to the user would be the raw Postgres error, leaking schema information.

Fix: Validate format with a regex (`/^\d{2}:\d{2}$/`) and bounds-check hours (0-23) and minutes (0-59) before proceeding.

---

**[MINOR-4] `requireCustomer` auth guard does not check role**

File: `src/lib/auth.ts:19-23`

```typescript
export async function requireCustomer() {
  const profile = await getProfile()
  if (!profile) redirect('/auth/login')
  return profile
}
```

This function is named `requireCustomer` but accepts any authenticated user regardless of role. A `cleaner` or `admin` user who calls a customer-only endpoint will pass this guard. Depending on business logic, this may be intentional (cleaners can also make bookings), but the naming is misleading and could cause confusion.

Fix: Either rename to `requireAuth()` if any role is acceptable, or add: `if (profile.role !== 'customer') redirect('/auth/login')`

---

**[MINOR-5] Middleware provides no centralised route-level protection**

File: `src/middleware.ts`

The middleware only calls `updateSession` — it does not block unauthenticated requests from reaching `/admin/*` or `/provider/*` routes. Protection is implemented individually per layout file and per route handler. This is correct today but creates risk of oversight when adding new routes.

Fix: Add path-based redirects in `middleware.ts` for known protected prefixes, as a safety net:
```typescript
if (request.nextUrl.pathname.startsWith('/admin') && !user) {
  return NextResponse.redirect(new URL('/auth/login', request.url))
}
```

---

**[MINOR-6] Admin cleaner status PATCH does not validate `status` input**

File: `src/app/api/admin/cleaners/[cleanerId]/route.ts:26-32`

```typescript
const { status } = await request.json()
const updateData: Record<string, unknown> = { status }
```

The `status` value from the request body is passed directly to the DB update without validation against the `cleaner_status` ENUM (`pending`, `approved`, `suspended`). Supabase/Postgres will reject an invalid enum value, but the raw error message is returned to the client at line 39 (`{ error: error.message }`), leaking Postgres schema details.

Fix: Validate `status` against the known enum values before executing the update.

### Priority Fix Order

1. **MAJOR-2** — Add RLS to `platform_settings` immediately. This is a one-line SQL fix with no code changes.
2. **MAJOR-1** — Restrict `bookings_update` policy. Financial field self-modification is a real attack vector.
3. **MAJOR-3** — Wrap ECPay notify writes in a transaction. Partial payment state is a data integrity issue.
4. **MAJOR-4** — Block suspended cleaners in API routes. One-liner check.
5. **MINOR-1** — Fix review update policy to prevent cleaners editing customer comments.
6. **MINOR-2** — Replace `Math.random()` with `crypto.randomBytes` in booking number generation.
7. **MINOR-3** — Validate `scheduledStartTime` format before arithmetic.
8. **MINOR-5** — Add middleware-level path protection as defence-in-depth.
